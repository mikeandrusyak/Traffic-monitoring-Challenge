Project Overview

This project demonstrates a production-grade traffic monitoring pipeline designed for real-time vehicle detection and post-acquisition data cleaning. The system captures raw video frames on a low-power Raspberry Pi, applies adaptive background subtraction to detect vehicles, tracks them via centroid matching, and stores results to a cloud database. Downstream processing applies sophisticated categorization and merging algorithms to clean up fragmented detections and produce reliable, deduplicated vehicle counts.

Project team: Mykhailo Andrusiak, Daniel Chekmenov, Artom Nerivnia, and Fiorenzo Luethi.

Reproducibility: Source code and raw data sample are publicly available.

The Problem

Real-world traffic monitoring from low-cost cameras (Raspberry Pi + module-level camera) faces several challenges that limit the reliability of raw detection outputs:

  • Fragmentation: A single vehicle passing through the frame may be split into multiple detection tracks due to occlusions, lighting changes, or temporary loss of tracking continuity.
  • Noise: Shadows, reflections, wind-blown debris, and camera artifacts generate false positives that inflate vehicle counts.
  • Variable visibility: Day/night transitions, weather changes, and camera movement degrade detection quality without adaptive algorithms.
  • Tracking drift: Simple centroid tracking (lowest-cost approach) can lose vehicles in crowded scenes or when objects overlap.

Goal: Build a post-processing pipeline that transforms noisy raw detections into a clean, deduplicated count of actual vehicle passages, suitable for traffic analysis (e.g., vehicle counts by time-of-day, compliance analysis, flow studies).

System Design

Architecture Overview

The system is decomposed into two main stages: Real-Time Capture (on Raspberry Pi) and Post-Acquisition Processing (batch cleanup).

1. Real-Time Capture Pipeline (main_cam_2.py)

Runs continuously on the Raspberry Pi with minimal CPU overhead:

  • Frame acquisition via Picamera2 at 640×480 YUV420 resolution
  • Contrast enhancement using CLAHE (Contrast Limited Adaptive Histogram Equalization)
  • Adaptive background subtraction (day mode: MOG2 with history=600, varThreshold=32; night mode: history=1500, varThreshold=18)
  • Morphological cleanup (5×5 elliptical kernel) and bus-merging kernel (20×20 rectangular closing)
  • Area filtering (800–40,000 pixels) to exclude noise and partial objects
  • Centroid tracking: Euclidean distance matching with 35-pixel threshold for ID continuity
  • Non-blocking async write: records batched to Google Cloud SQL via producer-consumer queue

2. Data Processing Pipeline (data_pipeline.py)

Batch processing that cleans and consolidates raw detections:

  • Session segmentation: Groups frames by continuity to identify separate recording sessions
  • Track categorization: Classifies each raw detection into 7 categories based on path completeness, size stability, and movement efficiency
    • Perfect: Clean, stable, >95% of ROI traversed
    • Partial: Good quality, 30–95% path, size stable
    • Merged: Result of fragment consolidation
    • Noise/Ghost/Static/Flickering: rejected due to insufficient evidence or instability
  • Fragment merging: Identifies vehicle fragments and merges them by matching temporal gaps (≤1.5s), spatial proximity (<40px), size similarity (<20%), and direction consistency
  • Unified ID assignment: Consolidates merged fragments into a single vehicle record
  • CSV export: Saves processed records with quality labels and unique IDs

Data Quality Result

19.1% of raw detections are accepted as high-quality tracks (Perfect + Partial + Merged = 103,282 records). Another 6.27x deduplication factor is achieved through fragment merging: 540,786 raw fragments are consolidated into 86,276 unique vehicles. This aggressive filtering ensures downstream traffic analysis relies only on reliable signals.

Fragment Merging Strategy

The merge algorithm identifies candidate pairs of detection fragments that likely represent the same vehicle interrupted by tracking loss or temporary occlusion. Pairs are matched by:

  • Temporal gap: Time between end of first fragment and start of second ≤ 1.5 seconds
  • Spatial proximity: Y-position difference < 40 pixels (accounting for vehicle movement)
  • Size similarity: Width difference < 20% (same vehicle should have consistent bounding box)
  • Direction consistency: Both fragments moving in same general direction (Y-axis)
  • Lane consistency: X-position in similar range (same lane)

Merge chains are built transitively: if fragment A matches B, and B matches C, all three are consolidated into a single vehicle record assigned a unified_id.

Limitations & Scope Boundaries

This project demonstrates a functional traffic monitoring system with solid engineering practices, but several limitations should be aware of for production deployment or competitive benchmarking:

Validation & Accuracy

No ground-truth validation: The categorization and merging algorithms have not been validated against manually-labeled data. Acceptance rate (19.1%) is theoretically derived but not empirically verified against labeled fraud/noise sets. Precision, recall, and F1 scores are unknown.

No benchmark comparisons: The centroid tracker is simple and effective for this use case, but no comparison to industrial trackers (YOLO + DeepSORT, Kalman-based systems) is provided. Unknown if accuracy is competitive at higher vehicle densities.

Operational & Environmental

Single camera, fixed location: System is designed and tested for a single Raspberry Pi with fixed camera positioning. Scalability to multi-camera setups or mobile cameras not yet explored.

Limited weather/occlusion testing: Monitoring period included only normal weather conditions. Robustness to heavy rain, snow, high wind, or prolonged occlusions (e.g., parked trucks blocking view) is untested.

Window blind effect (documented in field operation): A significant share of rejected trajectories was caused by moving office blinds in front of the scene. Wind-induced oscillation introduced persistent foreground noise and partial visual obstruction, especially increasing short-lived ghost tracks.

Camera calibration & perspective: Physical dimensions (height, width in pixels) are raw bounding-box measurements without perspective correction or camera intrinsic calibration. True vehicle sizes cannot be recovered without additional calibration.

Performance & Scalability

No throughput benchmarks: FPS, CPU usage, memory footprint, and latency on Raspberry Pi are not documented. Unknown if system meets real-time requirements under peak load (e.g., 60 vehicles/minute).

Database I/O not stress-tested: Async writes to Google Cloud SQL are efficient for current volume (~540K records over 40 days = ~13.5K per day = ~0.16 per second average). Scalability to 100+ FPS or multiple concurrent cameras unknown.

Algorithm Tuning

Hyperparameters not ablated: Key thresholds (distance_per_frame=35px, time_gap_limit=1.5s, width_similarity=20%) are chosen based on domain intuition, not systematic tuning. Sensitivity analysis not performed.

Future Work Recommendations:
  • Label ground-truth sample (100–500 vehicles) and compute precision/recall metrics
  • Compare centroid tracker to YOLO+DeepSORT on same video sequences
  • Stress-test multi-camera ingestion with simulated 30+ FPS streams
  • Perform hyperparameter sensitivity analysis (sweep distance, time, similarity thresholds)
  • Develop real-time camera calibration module for perspective-aware measurements
  • Test system robustness under adverse weather (rain, fog, night with poor lighting)

Technical Stack & Architecture

Hardware

  • Raspberry Pi 4/5 – Low-power compute platform for edge deployment
  • Pi Camera Module 3 – 12MP, 120° FOV, auto HDR, good low-light performance
  • Picamera2 driver – Native Python API for YUV420 frame capture

Software Stack

  • Python 3.9+ – Main language; flexible, well-supported on Pi
  • OpenCV 4.x – MOG2 background subtraction, morphological operations, tracking utilities
  • NumPy / Pandas – Efficient array operations and data processing
  • Google Cloud SQL + pg8000 – Cloud-hosted PostgreSQL for durability; non-blocking Python client for efficient async writes
  • google-cloud-sql-connector – Manages Cloud SQL IAM authentication and connection pooling
  • Plotly / Matplotlib – Visualization of detections, merges, statistics
  • Jupyter – Interactive notebooks for exploratory analysis and algorithm validation

Design Patterns

  • Producer-Consumer Queue: Real-time frame processing (producer) decoupled from database writes (consumer) via thread-safe FIFO queue
  • Adaptive Algorithms: MOG2 parameters switch between day/night based on frame brightness
  • Modular Pipeline: Capture, detection, tracking, and processing stages separated for testability and reuse
  • Batch Processing: Post-acquisition stages (categorization, merging) operate on CSV snapshots for reproducibility

Key Algorithmic Components

Component Algorithm Purpose
Background Subtraction MOG2 (Mixture of Gaussians) Separate foreground (vehicles) from background (road, static objects)
Morphology Cleanup Elliptical kernel opening/closing Remove noise, merge bus fragments, smooth boundaries
Object Tracking Centroid Euclidean Distance Matching Assign consistent IDs to objects across frames (simple, low-CPU)
Track Categorization Rule-based heuristics (path completeness, size CV, movement efficiency) Classify raw detections into quality tiers; reject noise
Fragment Merging Multi-constraint matching + transitive closure Consolidate interrupted vehicle tracks; improve vehicle count accuracy

Code Organization

  • main_cam_2.py – Real-time capture loop; frame acquisition, detection, tracking, async DB writes
  • tracker2.py – Centroid tracker class; ID management and matching logic
  • data_pipeline.py – Orchestrator for post-acquisition processing; loads CSV, applies categorization/merging, exports results
  • utils/transformer.py – Core logic for classification, merging, and consolidation
  • utils/loader.py – Database connector; loads raw detections from Google Cloud SQL
  • utils/save_to_database.py – Async writer class; producer-consumer batching for efficient multi-row INSERT
  • utils/plots.py – Visualization helpers (trajectory heatmaps, merge grids, distributions)
  • notebooks/raw_data_analysis.ipynb – Exploratory analysis; algorithm validation and parameter tuning
  • notebooks/summary_data_analysis.ipynb – Processed data analysis; statistics, trends, insights
Architecture Strengths
  • Low-power edge deployment: Raspberry Pi can run continuously with minimal overhead
  • Async durability: Non-blocking database writes ensure frame capture is never stalled by I/O
  • Reproducible batch processing: Post-acquisition stages can be re-run without re-capture
  • Adaptive algorithms: Day/night switching and heuristic-based categorization handle real-world variability
  • Modular design: Easy to test, validate, and extend individual pipeline stages

Conclusion

This Traffic Monitoring Challenge demonstrates end-to-end systems engineering applied to a real-world computer vision problem. The project combines hardware efficiency (Raspberry Pi), adaptive algorithms (day/night MOG2, heuristic categorization), cloud integration (Google Cloud SQL), and rigorous data quality assurance (categorization + merging) to extract reliable traffic metrics from noisy sensor data.

The 40-day dataset of 540K+ detections validates the system's ability to capture temporal patterns, reveal congestion hotspots, and measure compliance. With documented limitations and clear paths for enhancement (ground-truth validation, benchmarking, multi-camera scaling), this project serves as both a functional tool and a foundation for further research into traffic analysis and edge AI.
